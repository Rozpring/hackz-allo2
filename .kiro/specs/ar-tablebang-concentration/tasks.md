# Implementation Plan

> 凡例: `(P)` は先行のFoundation完了後に並行実装可能なタスク。`_Depends:_` は順序から自明でない依存のみ明記。

- [ ] 1. プロジェクト基盤とパラメータ・共通部品の整備
- [ ] 1.1 Xcodeプロジェクト作成とiOS/カメラ権限の初期設定
  - iOSアプリのXcodeプロジェクトを作成し、ARKit/RealityKit/Vision を利用可能にする
  - `Info.plist` にカメラ利用目的文（`NSCameraUsageDescription`）を追加する
  - ビルドが通り、空のARView画面が起動して背面カメラ映像が表示されることを確認できる
  - _Requirements: 10.2, 10.6_
- [ ] 1.2 調整パラメータの集約（GameConfig）
  - 速度しきい値・クールダウン・威力レンジ・影響半径カーブ・物理係数（質量/摩擦/反発）・静止しきい値・盤面サイズ・カード物理寸法/格子間隔/外周余白・制限時間・コンボ倍率を1箇所に集約する
  - デフォルト値（たたき台）を定義し、コードから容易に変更できる
  - 各値が型付きで参照でき、ビルドが通ることを確認できる
  - _Requirements: 4.3, 5.2, 8.5, 10.5_
- [ ] 1.3 座標変換ヘルパとイベント型の整備
  - 端末向きからカメラ画像orientationを算出、`ARFrame.displayTransform(for:viewportSize:)` を用いた Vision正規化座標→ARView画面座標への変換（回転・アスペクト・クロップを含む。単純y反転では不可）、画面座標→AR平面への投影前処理、カードのワールド上方向ベクトルから表/伏せを判定するヘルパを実装する
  - 台パンイベント・手サンプル等の共通イベント型を定義する
  - 各ヘルパの純関数部分が単体で呼び出せ、向き別の座標変換結果が期待通りであることを確認できる
  - _Requirements: 3.3_

- [ ] 2. ARセッションと権限・平面検出の土台
- [ ] 2.1 カメラ権限フローと拒否時の導線
  - 初回起動でカメラ権限を要求し、許可ならAR開始、拒否なら機能不可メッセージと設定アプリへの導線を表示する
  - 権限拒否時は盤面生成へ進まないことを画面上で確認できる
  - _Requirements: 1.1, 1.2_
- [ ] 2.2 ワールドトラッキングと水平面検出・状態通知
  - 水平面検出付きでARセッションを開始し、トラッキング状態と「平面検出可否」を購読可能にする
  - 平面未検出時はガイド、十分な広さの平面検出で配置可能を提示する
  - 実機でテーブルにかざすと平面が認識され配置可能表示に切り替わることを確認できる
  - _Requirements: 1.3, 1.4, 1.5, 10.3_
  - _Depends: 1.3_
- [ ] 2.3 台パン位置のスクリーン→AR平面投影
  - 画面座標をAR平面のワールド座標へ投影（raycast）する機能を提供する
  - 任意の画面点が平面上の妥当なワールド座標に変換され、平面外ではnilになることを確認できる
  - _Requirements: 4.4_
  - _Boundary: ARSceneController_
  - _Depends: 2.2_

- [ ] 3. カードと盤面（Cardsドメイン）
- [ ] 3.1 カード1枚の物理エンティティ (P)
  - 薄い箱の剛体カードを生成し、表面をローカル上方向、裏面を下方向に固定、ランクと状態（伏せ/表/回収）を保持する
  - 質量・摩擦・反発を設定し、重力で平面に落ち着くこと、上方向ベクトルの符号で表/伏せが判定できることを確認できる
  - _Requirements: 2.2, 2.4, 5.4, 7.2_
  - _Boundary: CardEntity_
- [ ] 3.2 各ランク2枚ずつのデッキ構築 (P)
  - 各ランクがちょうど2枚ずつのデッキを生成し、必ずペア成立可能なランク割当を行う
  - 指定ペア数に対し総枚数=2×ペア数、各ランクが2枚である不変条件をテストで確認できる
  - _Requirements: 2.3, 2.4_
  - _Boundary: DeckFactory_
- [ ] 3.3 盤面生成・格子配置・床/壁・回収管理
  - アンカー配下にデッキを格子状・全伏せで配置し、静的な床と外周の不可視壁を設置、中心からの半径内カード抽出・回収除去・残ペア数管理を提供する
  - デバッグモードで平面タップなしの自動配置に対応する
  - 盤面が現実空間にアンカー固定され、端末が揺れても位置が保たれること、回収でカードが消え残ペア数が減ることを確認できる
  - _Requirements: 2.1, 2.5, 2.6, 5.1, 5.8, 6.6, 7.1, 7.3, 7.5_
  - _Depends: 3.1, 3.2, 2.2_

- [ ] 4. 手検出と威力（Inputドメイン）
- [ ] 4.1 手検出プロバイダの抽象と手サンプル型 (P)
  - カメラフレームから手の代表点（画面座標＋時刻＋信頼度）を供給する抽象インターフェースを定義する
  - モック実装で任意の代表点列を発行でき、購読側が受信できることを確認できる
  - _Requirements: 3.1, 3.2_
  - _Boundary: HandLandmarkProvider_
- [ ] 4.2 Visionによる手検出実装（別キュー・間引き・向き補正）
  - `ARFrame.capturedImage` をVisionの手姿勢検出に渡し、中指MCP（手のひら中心相当）を代表点として供給する
  - 専用キューで推論し前フレーム処理中はスキップ（間引き）、端末向き補正と信頼度フィルタを行い、低信頼度は未検出扱いにする
  - 実機で手をかざすと代表点が安定して流れ、手をフレーム外に出すと発行が止まることを確認できる
  - _Requirements: 3.1, 3.4, 3.5, 10.1, 10.6_
  - _Depends: 4.1, 1.3_
- [ ] 4.3 振り下ろし速度算出と台パン成立判定
  - 代表点の時系列から下方向速度を実測Δtで算出し、振り下ろしのピーク速度を保持、急減速（着地）で台パン成立イベントを発行する
  - クールダウンで過剰連打を抑制し、手未検出中は判定しない
  - 合成サンプル列で、しきい値超の振り下ろし→急減速時のみ成立し、クールダウン中・手未検出では成立しないことをテストで確認できる
  - _Requirements: 3.3, 4.1, 4.5, 4.6_
  - _Depends: 4.1_
- [ ] 4.4 ピーク速度→威力の正規化 (P)
  - ピーク速度を最小〜最大威力にクランプ正規化する
  - 速度増加に対し威力が単調増加し、範囲外がクランプされることをテストで確認できる
  - _Requirements: 4.2, 4.3_
  - _Boundary: PowerCalculator_
  - _Depends: 1.2_

- [ ] 5. 衝撃波と物理挙動（Physicsドメイン）
- [ ] 5.1 衝撃波による距離減衰インパルス付与
  - 威力から影響半径を決め、半径内の各カードへ距離減衰インパルスを与える。重心からずらした打点と軸ランダムトルク・微小ランダム並進で非決定論的な跳ね・回転を生む
  - 強い台パンほど半径が広がりめくれ得る枚数が増えること、結果が毎回ばらつく（浮くだけ/反転/同面/複数回転）ことを実機で確認できる
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Depends: 3.3, 1.2_
- [ ] 5.2 静止検出と表裏確定・疑似スリープ
  - 各カードの速度を監視し、しきい値以下が連続したフレームで静止とみなし、上方向ベクトルの符号で表/伏せを確定して「盤面静止」を通知する。静止カードは疑似スリープ化し台パン時に復帰、物理ロスト時もクラッシュしないフォールバックを設ける
  - 台パン後にカードが落ち着くと状態が確定して通知が出ること、表のカードが後続台パンで伏せに戻り得ることを確認できる
  - _Requirements: 5.6, 7.3, 7.4, 10.4_
  - _Depends: 3.1, 3.3_

- [ ] 6. ゲームロジック（Gameドメイン）
- [ ] 6.1 表カードの同一ランクペア自動検出 (P)
  - 物理静止後、表になっている全カードを走査し、同一ランクが2枚揃う組をすべて検出する。相手が表に揃わないカードは未成立として残す
  - 複数ペア同時・単独成立・未成立混在の入力に対し、期待するペア集合のみ返すことをテストで確認できる
  - _Requirements: 6.1, 6.2, 6.3, 6.5, 7.4_
  - _Boundary: MatchEvaluator_
- [ ] 6.2 スコア・コンボ・残カード・制限時間・勝敗管理
  - 盤面静止通知を受けてペアを回収・加点し、1回で複数ペア成立時はコンボ倍率を上げる。残カード・スコアを更新し、制限時間カウントと全クリア／タイムアップを判定する
  - 全ペアクリアでクリア、時間切れでタイムアップに遷移し、スコア・コンボが期待通り増えることを確認できる
  - _Requirements: 6.3, 6.4, 6.6, 8.1, 8.2, 8.3_
  - _Depends: 6.1, 5.2, 3.3_

- [ ] 7. UIと触覚/音
- [ ] 7.1 プレイ中HUD (P)
  - 残り時間・現在スコア・直近の威力（ゲージ/数値）・コンボ状況を表示し、手未検出時に補助ガイドを出す。AR盤面の視認を妨げない配置にする
  - プレイ状態の変化に応じてHUD表示が更新されることを確認できる
  - _Requirements: 9.1, 9.2, 9.3, 9.4, 9.5_
  - _Boundary: HUDView_
  - _Depends: 6.2_
- [ ] 7.2 結果画面とリトライ (P)
  - クリア時は経過時間とスコア、タイムアップ時は残ペア数とスコアを表示し、再プレイ導線を提供する
  - 各終了状態で適切な結果が表示され、リトライで初期状態に戻れることを確認できる
  - _Requirements: 8.2, 8.3, 8.4_
  - _Boundary: ResultView_
  - _Depends: 6.2_
- [ ] 7.3 触覚・効果音フィードバック (P)
  - 台パン・カードめくり・ペア成立に視覚エフェクト・触覚・効果音を付与する
  - 各イベントで触覚/音が再生されることを実機で確認できる
  - _Requirements: 5.7_
  - _Boundary: FeedbackController_

- [ ] 8. 結線（Integration）
- [ ] 8.1 カメラフレームの二系統分配
  - ARセッションのフレームを、AR描画/物理と手検出プロバイダの双方へ供給する結線を行う
  - 1つのカメラフィードでAR表示と手検出が同時に動作することを実機で確認できる
  - _Requirements: 3.1, 10.1_
  - _Depends: 2.2, 4.2_
- [ ] 8.2 台パンから得点までのエンドツーエンド結線
  - 台パン成立→威力算出→画面座標を平面投影→衝撃波→物理→静止確定→ペア検出→回収/加点/HUD更新、までを1本のフローとして接続する
  - 実機で台パンするとカードがめくれ、揃ったペアが自動回収されスコアが増える一連が動作することを確認できる
  - _Requirements: 4.4, 5.1, 6.2, 7.1_
  - _Depends: 4.3, 4.4, 5.1, 5.2, 6.2, 2.3, 7.1_
- [ ] 8.3 画面遷移（権限→配置→プレイ→結果）
  - 権限・平面配置・プレイ・結果の画面状態遷移を統合し、配置完了で制限時間カウントを開始する
  - 起動から結果表示・リトライまで一連の遷移が破綻なく動作することを確認できる
  - _Requirements: 8.1, 8.4_
  - _Depends: 2.1, 3.3, 7.1, 7.2_

- [ ] 9. 検証（Validation）
- [ ] 9.1 ユニットテスト整備
  - 台パン成立判定・威力正規化・ペア検出・デッキ不変条件・衝撃波の距離減衰純関数を対象にテストを実装する
  - 主要ロジックのテストが緑になることを確認できる
  - _Requirements: 3.3, 4.1, 4.2, 4.3, 4.5, 4.6, 6.1, 6.3, 6.5, 2.3, 2.4, 5.2, 5.3_
  - _Depends: 4.3, 4.4, 6.1, 3.2, 5.1_
- [ ] 9.2 統合テスト（モック手検出の連鎖）
  - モック手検出→速度→威力→台パンイベントの連鎖、盤面静止→ペア検出→スコア更新の連鎖を検証する
  - 結線レベルのテストが緑になることを確認できる
  - _Requirements: 6.2, 6.4, 7.4, 8.1_
  - _Depends: 8.1, 8.2_
- [ ] 9.3 実機E2E・性能計測
  - 起動→権限→平面検出→配置→台パン→めくれ→ペア回収→全クリア/タイムアップ→リトライの主要動線を実機で確認する
  - ARKit＋手検出＋16枚物理の同時稼働で滑らかなフレームレートを維持すること、強/弱の台パンで影響半径とめくれ枚数が変化すること、表カードが後続台パンで伏せ得ることを計測・確認できる
  - _Requirements: 1.1, 1.2, 1.5, 5.2, 7.3, 8.2, 8.3, 8.4, 10.1, 10.3, 10.4_
  - _Depends: 8.2, 8.3_
